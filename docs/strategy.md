# Portfolio Strategy

## 1. Sitemap

- `/` - homepage with hero, engineering impact, selected products, architecture highlights, case studies, open-source/GitHub metrics, timeline, and contact
- `/case-studies/ai-customer-support-platform` - AI support architecture brief
- `/case-studies/multi-tenant-saas-starter` - SaaS tenancy architecture brief
- `/case-studies/workflow-automation-engine` - automation engine architecture brief
- `/api/github` - cached GitHub REST snapshot for repository/profile metrics
- `/sitemap.xml` - generated by Next metadata route
- `/robots.txt` - static crawler policy

## 2. UX Strategy

Target users are technical founders, YC-style startup teams, engineering managers, and remote-first SaaS companies. The page should answer three questions quickly:

- Can this engineer reason about production SaaS systems?
- Is the GitHub profile credible enough to inspect further?
- Which repositories should I open first?

The homepage avoids a resume timeline as the primary artifact. It behaves more like a product architecture review: system surfaces, metrics, repository triage, and concise case-study cards.

## 3. Design System

- Dark mode default with enterprise SaaS contrast
- Restrained navy/graphite background, teal primary signal, blue and amber secondary accents
- 8px card radius, thin borders, dense dashboard surfaces
- Inter typography with balanced headings and compact product copy
- Motion only for reveal, metric emphasis, and dashboard micro-interactions
- No decorative blobs, hero illustrations, or freelancer-style service blocks

## 4. Visual Direction

The strongest source patterns extracted from research:

- Brittany Chiang: concise hero and craft, but avoid the over-copied left-nav portfolio template
- Josh Comeau: use case-study storytelling rather than only project thumbnails
- Lee Robinson and Vercel: high-signal personal positioning, simple navigation, developer trust
- Linear: customer-story style metrics, category filters, and crisp product proof
- Stripe: dense technical language, product taxonomy, and operational confidence
- Dub.co: outcome-oriented dashboards and “measure what matters” framing
- Cal.com: infrastructure-level product positioning instead of feature clutter
- Trigger.dev: workflow/run status visuals, realtime execution, and developer-first architecture surfaces

## 7. Database Schema

No database is needed for the current portfolio. GitHub data can be fetched from the public REST API and cached with Next.js `revalidate`.

If the site later stores private analytics or manual repository assessments, use:

```sql
create table repository_assessments (
  id uuid primary key,
  repo_name text not null,
  visibility text not null default 'public',
  narrative_score integer not null,
  evidence_score integer not null,
  recommendation text not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

## 8. GitHub Integration Strategy

- Use the public REST API for profile and repository metrics
- Cache `/api/github` for one hour
- Do not require a token for the public homepage
- Keep strategic recommendations static so they do not fluctuate with API rate limits
- In the future, add optional `GITHUB_TOKEN` on Vercel for higher rate limits and contribution graph queries

Profile repair recommendations:

- Unpin `Ghost`, `strapi`, and `django` unless there are visible pull requests or fork-specific branches worth reviewing
- Pin original, implemented systems only
- Add descriptions, topics, screenshots, deployments, and issue roadmaps to every highlighted repo
- Rename or hide empty placeholder repos until they contain code
- Add location, website, and cleaner bio copy to GitHub profile

## 9. Deployment Strategy

- Deploy to Vercel as a Next.js project
- Set production domain to the GitHub Pages URL only if DNS/custom domain is configured; otherwise use Vercel URL and link it from GitHub profile
- Keep GitHub Pages repo as the source repo name if desired, but use Vercel for Next.js SSR/API support
- Run `npm run build` before deployment
- Add Vercel project environment variable only if using an authenticated GitHub token later

## 10. Implementation Plan

1. Convert static site to Next.js 15 App Router
2. Add TypeScript, Tailwind, Framer Motion, shadcn-style primitives, and MDX
3. Build homepage sections around architecture and product proof
4. Add MDX case-study pages for the strongest systems
5. Add GitHub API integration and current profile audit
6. Add SEO metadata, Open Graph image, sitemap, robots, and Vercel config
7. Verify mobile/desktop layout, console, typecheck, and production build
8. After deployment, update GitHub profile URL and pinned repositories
